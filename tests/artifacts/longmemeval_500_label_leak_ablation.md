# LongMemEval v2 Label-Leak Ablation

The rejected `v2` patch used dataset-specific structure embedded in LongMemEval support IDs. In [src/waggle/longmemeval_benchmark.py](/Users/abhigyanshekhar/Desktop/MCP/src/waggle/longmemeval_benchmark.py), it matched session IDs with a regex shaped like `^answer_([A-Za-z0-9]+)(?:_abs)?_\\d+$`, extracted the family token from IDs such as `answer_593bdffd_4`, and promoted all siblings from the same family when one sibling appeared among the top raw candidates. That is not production graph retrieval; it is reading benchmark naming structure directly from the support ID.

Those leaked numbers looked dramatically better than the honest baseline. On the 500-case graph-raw report, overall `Exact@5` jumped from `76.2%` to `93.6%`, while `R@5` rose from `92.6%` to `94.8%`. The most suspicious part was the cardinality behavior: support-cardinality `3`, `4`, and `5` all moved to exactly `100.0% Exact@5`, even though the original run had severe cliffs there (`36.6%`, `26.3%`, and `9.1%` respectively). When sibling promotion was disabled for `answer_*` IDs, the numbers collapsed back to the original v1 report, confirming that the gain came from parsing the benchmark’s gold-ID convention rather than from a general retrieval improvement.

The patch was rejected because it would not transfer to LoCoMo, production conversational memory, or any dataset that does not happen to encode answer-family membership in its node IDs. The production-safe replacement is the `memory_benchmark.py` indexing approach: represent each session as a session-anchor node, attach turn nodes to that anchor with graph edges, connect nearby turns and adjacent sessions structurally, and let retrieval exploit only signals that exist in real data such as shared `session_id`, graph topology, temporal adjacency, and semantic similarity. That path is slower and so far less impressive numerically, but it is the only publishable direction because it does not depend on benchmark-specific label structure.
